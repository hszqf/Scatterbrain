# Agent 易误解点记录（队列表现专项）

> 目的：避免后续再次把“表现需求”误实现成“数据改动”。

## 1) “只改表现，不改数据语义”
- 用户明确说“都是表现，不是数据”时：
  - 只能改 `scripts/game/` / `scenes/game/` 的视觉与播放流程；
  - 不得改动 `core` 编译语义、队列真相、变化解释规则。

## 2) 新变化的状态图只能生成一次
- 正确表现：新变化的视觉 badge 从来源飞入队列区域后，继续作为同一个视觉对象参与入列挤压。
- 错误表现：先飞入后消失，再在最左侧重新生成一个 badge（二次生成）。

## 3) 入列挤压阶段的同步关系
- 当新变化进入格子（slot）时，已有变化必须**同速**向右移动，像被挤出去。
- 右移到位后，才执行被挤出项的放大/淡出离场。

## 4) 不要把“实现细节”当“用户要求”
- 用户要的是体感与可见节奏，不是指定具体节点实现。
- 可自由安排内部数据/缓存方式，但必须保证最终视觉行为与需求一致。

## 5) 测试沟通要真实
- 不得声称“测试全部通过”，除非命令真实通过。
- 若失败，必须给出失败命令与关键错误摘要（如 harness summary 与 exit error）。

## 6) 以后 PR 描述必须先写“用户原始需求”
- PR 描述第一段先原文/近原文复述用户要求（先写需求，再写实现）。
- 禁止直接从“我做了什么”开头，避免偏离用户意图。
